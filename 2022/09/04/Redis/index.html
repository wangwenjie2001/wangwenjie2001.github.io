Redis

发表于2022-09-04|更新于2022-09-05|后端
|字数总计:5k|阅读时长:18分钟|阅读量:

Redis（Remote Dictionary Server )，即远程字典服务，是一个开源的使用ANSI C语言编写、支持网络、可基于内存亦可持久化的日志型、Key-Value数据库，并提供多种语言的API。

一、Redis概念

NoSQL非关系型数据库

减少IO压力、直接可以作为缓存使用（减少数据库的压力）

适用于高并发和海量数据操作

1.1    Redis安装

1、安装GCC

yum install gcc安装

gcc --version测试

2、解压redis压缩包

tar -zxvf redis06.2.7.tar.gz

3、进入Redis目录make命令编译

4、安装Redismake install

5、切换目录到 cd /user/local/bin

Redis-server： 服务器启动命令

Redis-cli：客户端，操作入口

1.2    启动

前台启动：redis-server

后台启动：

1、将redis的核心配置文件赋值一份cp redis.conf /etc/redis.conf

2、修改核心配置文件中的daemonize为yes

3、进入 cd /user/local/bin使用redis-server /etc/redis.conf

4、进程查看 ps -ef | grep redis

5、redis-cli启动客户端

6、关闭Redis：shutdown

1.3    基本概念

端口：6379

技术：单线程+多路IO复用

默认16个数据库，默认0号数据库

二、五大数据类型

2.1    Key操作

查看所有key： keys *

判断某个key是否存在： exists key值

查看类型：type key值

删除指定key：del key值

非阻塞删除key：unlink key值(异步操作)

设置key的过期时间：expire key值 秒数 -1表示永不过期、-2表示过期

查看还有多少秒过期：ttl key值

切换数据库：select 数据库号

查看当前库有多少数量的key：dbsize

清空当前库：flushidb

清空所有库：flushall

2.2    字符串

字符串最多可以是512M

添加键值对：set key value

获取键值对：get key value

设置相同的key会被覆盖

追加原值：append key value

获取值的长度：strlen key

添加键值对（只有key不存在才会设置成功）： setnx key value

使得数据值+1：incr key(数据类型)

使得数据值-1：decr key(数据类型)

数据值+自定义：incrby/decrby key 自定义值

设置多个键值对：mest key1 value1 key2 value2

获取多个键值对：mget key1 key2

添加多个键值对（只有key不存在才会设置成功）：msetnx key1 value1 key2 value2有任何一个不存在都会失败

获取值的范围：get range key startIndex endIndex

将值设置到指定范围：set range key index value

设置值的时候设置过期时间：setex key outdate value

新值换旧值：getset key value取到的还是旧值但数据库中已经替换成了新值

底层数据结构：动态字符串

2.3    列表List

单键多值

底层：双向链表

从左/右插入多个值：lpush/rpush key value1 value2 value3

从左/右取值：lrange key start stop(-1到尾部)左取值顺序会以头插法顺序取出

从左/右吐一个值：lpop/rpop key取值取完，键值也就不存在了

从key1列表右边吐出一个值插到key2列表的左边：rpoplpush key1 key2

根据索引下标取出指定的值：lindex key index

获取列表长度：llen key

在value后面插入newvalue：linsert before value newvalue

从左右删除n个value：lrem key n value将n个前相同value的值相同

将列表key下标为index的值替换成value：lset key index value

数据结构：快速链表

1、数据少：压缩列表

2、数据多：把多个压缩列表变成链表

2.4    集合Set

无序集合，底层是一个哈希表。

添加一个或多个元素：sadd key value value

获取该集合所有值：smembers key

判断集合中是否含有该value值：sismember key value有返回1没有返回0

返回集合的元素个数：scard key

删除集合中的某个元素：srem key value1 value2

随机从集合吐出一个值：spop key

随机从集合取出n个值：srandmember key n

把集合中的一个值从一个集合移动到另一个集合：smove 起始key 目标key 值

返回两个集合的交集元素：sinter key1 key2

返回两个集合的并集元素： sunion key1 key2

返回两个集合的差集元素：sdiff key1 key2(key1中有的，key2中没有的)

数据结构：字典，哈希表实现

2.5    哈希Hash

是一组field和value的映射表

key：user

value：field：id 、name、age

            value：1、wwj、20

添加：hset key field value

获取：hget key field

添加多个：hmset key field1 value1 field2 value2

判断field是否存在：hexists key field

列出该集合的所有field：hkeys key

列出该集合所有的value：hvals key

为哈希表中的ifleld的值增量1 -1：hincrby key field increment（第增量）

当前field不存在的时候设置值：hsetnx key field value

数据结构：压缩列表、哈希表

2.6    有序集合Zset

没有重复元素的字符串集合

评分机制类似于权重

添加：zadd key score value

获取：zrange key 0 -1 从小到大 加入withscores能将评分一起返回

取评分范围：zrangebyscore key min max

从大到小排序：zrevrangebyscore key max min

score增量：zincrby key 增量 value

删除该集合下的指定值元素：zrem key value

统计该集合，分数区间内的元素个数：zcount key min max

返回集合中的排名从分数0开始：zrank key value

数据结构：等价于Map集合，跳跃表（类似B树，二分）、Hash、

三、配置文件详解

1、任何单位都是以bytes为基本单位

2、限定本地连接

3、开启保护模式（远程访问）

4、tcp-backlog时tcp的一个连接队列，包含未完成三次握手和已经完成三层握手总和的队列

5、timeout：默认0，超时时间

6、tcp-keepalive：心跳检测连接

7、pidfile：保存进程端口号

8、loglevel：日志级别

9、logfile：设置文件输出路径

10、security设置密码

11、LIMITS：设置客户端的连接数等

12、maxmemory-samples：样本数量

四、发布和订阅

消息发布：发布消息

消息订阅：接收消息

Redis客户端可以订阅任意数量的频道

订阅频道：subscribe channel1

发送消息：publish channel1 hello返回数字是订阅的数量

五、新数据类型

5.1    Bitmaps

位操作

value只能放二进制数据

偏移量可以称为一个用户的id记录

添加：setbit key 偏移量 value

取值：getbit key 偏移量

统计：bitcount key index1 index2 为1的总和数量，可以取index范围

操作 and or ：setbit and 合并后的key key1 key2

5.2    HyperLogLog

基数：不重复的元素集

添加：pfadd key element

计算基数数量：pfcount key

合并：pfmerge newkey key1 key2将key1和key2合并到newkey中

5.3    Geospatial地理类型

添加：geoadd key 经度 纬度 名字

获取坐标：geopos key 名字

获取两个位置的直线距离：geodsit key name1 name2 单位km

获取经纬度内的半径内的元素：georadius key 经度 纬度 1000单位km

八、事务操作

2.1    基本操作

串联多个命令执行，防止别的命令插队

Multi：类似于开启事务，组队阶段，输入的命令会依次进入命令队列

Exec：Redis会将之前输入的命令队列中的命令一次执行

discard：放弃组队****

在组件过程中命令发生失败，队列直接失败

在执行过程中命令发生失败，队列会执行失败之外的其余命令

命令：multi exec dicard

2.2    事务冲突

悲观锁：每次拿数据的时候都认为别人会修改数据，所以在每次拿数据的时候都会上锁，这样别人想拿这个数据就会被锁，直到它拿到锁为止。

乐观锁：每次拿数据都认为别人不会修改，不会上锁，但在更新的时候会判断一下在此期间别人有没有更新数据，使用版本号等方式判断。（适合读）

乐观锁使用：在执行multi之前先对key执行 watch key

当一个用户操作了里面的数据，另外一个用户再想进行操作就出现nil事务打断

unwatch 取消监视

事务的三大特性

1、单独的隔离操作：事务中的所有命令都会序列化，按顺序执行，事务在执行的过程中，不会被其他客户端的及命令请求所打断

2、没有隔离级别概念

3、不保证原子性：事务中如果有一条命令执行失败，后面的命令仍然会被执行

ab工具（模拟并发操作）：yum install httpd-tools

-n : 请求次数

-c : 并发次数

-t：提交数据类型POST请求请求地址

-p: 设置参数类型为content-type

九、数据持久化

Redis本身是一个内存数据库，可以使用RDB和AOF将数据存到硬盘（持久化操作）

2.1    RDB

在指定的时间间隔内将内存中的数据集快照写入硬盘（每隔一段时间获取当前时间的所有数据进行持久化操作）

但单独创建一个子进程（fork）来进行持久化操作，会先将数据写入到一个临时文件中，等待持久化过程都结束了，再用临时文件替换上次持久化文件（写时复制技术）。

save：在一个时间间隔内，有多少个key发生变化进行持久化操作 save seconds key

优势：适合大规模数据恢复

缺点：follk技术数据庞大造成性能影响，最后一次持久化操作可能丢失

2.2    RDB文件恢复

将RDB文件拷贝到rdb文件的工作目录

启动Redis，备份数据会自动加载

2.3    AOF

以日志的形式记录每个写操作，将所有写的指令写下来追加到文件中。

AOF和RDB同时开启会默认读取AOF的数据

异常恢复：修复AOF文件，通过 /usr/local/redis-check-aof –fix 命令加上修复文件名称，可以进行修复

AOF同步频率设置：

appendsync always ：始终同步，写入都会立刻记录到日志，性能较差完整性好

appednsync eversec：每秒同步，如果宕机，本秒数据会丢失

appendsync no： redis不主动进行同步，把同步时机交给操作系统

重写压缩操作：将多个指令合成一个指令（AOF当前大小>=baseSzie+baseSize * 100%）》=64mb执行重写操作）

AOF持久化流程：

1、客户端请求写命令被追加到AOF缓存区

2、AOF缓冲区根据AOF持久化策略将操作sync同步磁盘AOF文件

3、AOF文件超过重写策略或手动重写时，会被AOF文件重写，压缩AOF文件容量

4、服务重启会加载AOF文件恢复数据

十、主从复制

    主机数据更新后根据配置和策略，自动同步到备机的master/slaver机制，master写为主，slave以读为主

1、实现读写分离，增大性能

2、容灾（一个从的宕机可以快速去切换别的从）快速恢复

10.1    搭建主从

1、创建一个新的文件夹

2、将Redis.conf配置文件复制进来

3、复制多三个Redis核心配置文件分别启动

连接某个Redis服务：redis-cli -p port

进入客户端查询服务器状态信息：info repliation

配置从机：salveof 主机IP 端口号

10.2    一主两从，复制原理

一主二从：

1、当从服务器宕机之后再启动，从服务器失去从机的身份变成主机需要重新配置

2、当从服务器重新配置好从机的时候，数据和主机同步

3、当主服务器宕机之后，从服务器角色依然是从服务器，当主服务器重新启动的时候依然会回到主服务器的位置

复制原理：

1、从连接到主服务器之后会发送会向主服务器进行数据消息同步

2、主服务器会接到从服务器发过来的同步消息，把主服务器数据进行持久化，将rdb文件发送给从服务器，从服务器会进行读取

3、每次主服务器进行写操作之后会和从服务器进行数据同步

10.3    薪火相传，反客为主

薪火相传：

在从服务器中继续配置一个从服务器，解决主服务器同步压力

反客为主：

salveof non one：当主机宕机的时候，将从机变成主机（手动切换）

10.4    反客为主自动版

1、新建 sentinel.conf文件

2、配置内容(1：至少有多少个哨兵同意才切换主从）

3、启动哨兵：redis-sentinal 配置文件

复制延迟：当主服务器宕机了，需要将一些数据给同步给从服务器带来的延迟

主机宕机的时候之前的主机就会变成从机

11.3    缓存穿透

    当应用服务器压力突然增大的时候，导致访问Redis缓存获取不存在的数据，就会去查数据库导致数据库压力增大

问题原因：

1、redis查询不到数据库

2、出现非正常访问数据库（因为Redis没有会一直访问数据库）

解决方案：

1、对空值进行缓存

2、设置可访问名单（bitmap）

3、采用布隆过滤器

4、实时监控

11.4    缓存击穿

    数据库访问压力瞬时增加，redis并没有出现大量key过期，redis还是正常运行状态。

问题原因：

1、redis某个key过期，大量访问都使用这个key

解决方案：

1、预先设置热门数据到redis中

2、实时调整，调整过期时长

3、使用锁：排它锁

11.5    缓存雪崩

    数据库压力变大，服务器崩溃

1、在极少时间段，查询大量key的集中过期情况

解决方案：

1、构建多级缓存架构

2、使用锁或者队列

3、设置过期标志更新缓存

4、将缓存失效时间分散开

11.6    分布式锁

解决问题：操作一把锁对整个集群都有效

使用setnx特性（数据库内不能加入相同的key值除非删除）实现分布式锁（添加一个数据就是加锁，删除一个数据就是释放锁）

需要设置过期时间：防止一直被锁，其他请求无法访问

问题：服务器宕机无法设置过期时间：上锁同时设置过期时间

Lua脚本：支持原子性操作而且不会被打断
